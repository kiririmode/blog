# 流行りの仕様駆動開発は何を指しているのか

ここ一年ほどで、「仕様駆動開発」という言葉は急に市民権を得ました。GitHub は Spec Kit を通じて、仕様を中心に `/specify`、`/plan`、`/tasks` と段階化し、AI エージェントに実装させるプロセスを提示しています。そこでは仕様は「静的な文書」ではなく、プロジェクトとともに更新される living documents であり、AI と人間がともに参照する SSoT だと説明されています。。

[https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/:embed]

ただし、ここで問題が起きます。仕様駆動開発という言葉が広がる速度に対して、「仕様とは何か」「どこまでを仕様とみなすのか」という定義が追いついていません。Thoughtworks は、仕様駆動開発を「よく作られたソフトウェア要求仕様をプロンプトとして用い、AI コーディングエージェントで実行可能コードを生成する開発パラダイム」と説明する一方で、業界内には「仕様だけを保守すればよく、コードは中間生成物にすぎない」と考える急進的な立場と、「仕様はあくまでコード生成を駆動する要素であり、最終的に保守すべき SSoT は実行可能コードだ」と考える立場が共存していると整理しています。[Thoughtworks の Technology Radar](https://www.thoughtworks.com/radar/techniques/spec-driven-development) も、定義はまだ流動的だと述べています。

[https://www.thoughtworks.com/insights/blog/agile-engineering-practices/spec-driven-development-unpacking-2025-new-engineering-practices:embed] ￼

さらに、[https://arxiv.org/html/2602.00180v1:title]や[https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html:title]この曖昧さをより明示的に捉えています。そこでは、仕様駆動開発とは「仕様を SSoT とし、コードを生成または検証される二次成果物として扱う」発想だと定義したうえで、実際には spec-first、spec-anchored、spec-as-source という強度の異なる三つのレベルがあると整理しています。つまり、仕様駆動開発は単一の教義ではなくスペクトラムです。だから「spec -> plan -> tasks に進めば自動的に SDD である」とも言えませんし、「仕様を正確に書ければ人間はもうコードと向き合わなくてよい」とも言えません。まずこの言葉の揺れを自覚しないと、議論はすぐに宗教戦争になります。

あるあるですが、ぼくがいま仕様駆動開発に対してよくわからんなぁと思っているのは、道具が先に広まっているからです。もちろん、それ自体は悪いことでもないのですが。結果として、「とりあえず SDD のツールを入れればよい」「spec-only でいけるらしい」「人間はレビューだけしていればよい」という、魅力的だが雑な言説が出回ります。仕様駆動開発は、たしかに今の AI 開発に必要な方向を含んでいると思います。けれど、その“仕様”が何を担い、何を担えないかを先に整理しないと、話はすぐに危険な近道へ転んで、よくわからん議論に巻き込まれる。

# 仕様駆動開発の「仕様」とは何か

仕様駆動開発の議論をややこしくしているもう一つの理由は、「仕様」という言葉が、あまりにも多くのものを兼ねていることです。要件定義書を思い浮かべる人もいれば、PRDを思い浮かべる人もいる。設計方針、受入基準、テスト観点、作業分解、アーキテクチャ原則まで含める人もいます。GitHub の Spec Kit は、まさにその曖昧さを工程分割で吸収しようとしています。高レベルの要求と成功条件を仕様として起こし、それを技術的な実装計画に落とし、最後にタスクへ分解する。重要なのは、ここで「仕様」が一枚の神聖な文書ではなく、意図を徐々に具体化していく一連の構造化成果物だという点です。GitHub はこの各段階にチェックポイントを置き、「あなたの役割は方向を定めるだけではなく検証することだ」と明言しています。  ￼

Thoughtworks の整理もこれに近いです。spec-driven development は、単に要件を書く作法ではなく、構造化された機能仕様から複数の段階を経て、解決策やタスクへ落としていくワークフローだとされています。ただし同時に、それは opinionated であり、ツールやタスクの種類によって大きく変わるとも注意されています。つまり、「仕様」は名詞というより、AI に危険な補完をさせないためのコンテキスト設計そのものです。  ￼

Addy Osmani の “How to write a good spec for AI agents” は、この点をさらに実務的にしています。

[https://www.oreilly.com/radar/how-to-write-a-good-spec-for-ai-agents/:embed]

そこでは、良い仕様とは「AI を圧倒する巨大文書」ではなく、ちょうど必要な粒度の情報を与えるものだとされます。高レベルのビジョンから始め、AI に詳細計画を展開させ、必要な構造、テスト、境界条件、制約を盛り込みつつ、巨大な一枚岩ではなく扱える単位に分ける。そしてその仕様は、AI と人間が共同で育てる永続的なリファレンスであり、途中で何度でも書き換えてよい living document だとされます。仕様は AI に仕事をさせるための入口であると同時に、人間が意図を見失わないための成果物でもあるわけです。  ￼

だから、仕様駆動開発の「仕様」を一言で片付けるなら、AI に実装を委譲するために人間が先に外化しておくべき意図・制約・成功条件・境界条件の束です。 それは Word 一冊でも、Markdown 一枚でも、チケット群でもよい。大事なのは媒体ではなく、AI が勝手に危険な補完をしないために必要なだけ具体的であり、かつ人間が検証可能な構造を持っていることです。ここを外すと、「仕様」という看板の下に、単なる長文メモと擬似コードとレビュー観点の残骸が押し込まれていきます。

# spec-only は魅力的だが危険すぎる

仕様駆動開発の議論が盛り上がると、かなり高い確率で「なら、自然言語仕様を厳密に書ききれば、人間はコードを細かく読まなくてもよいのではないか」という誘惑が現れます。気持ちはわかるよ気持ちは。コードより先に仕様を整え、AI にそれを忠実に実装させられるなら、変更の中心は仕様に移り、レビューも仕様レビュー中心へ寄せられる。見た目はたいへん美しい。プロジェクトマネージャーも安心しそう。ガバナンス文書も増えます。実にそれっぽい。

しかし Addy Osmani の理解負債(Comprehension Debt)の記事が突いているのは、まさにその「それっぽさ」の危険です。理解負債とは、システム内に存在するコード量と、それを人間が本当に理解している量のギャップです。AI がコードを大量に生成できるようになると、このギャップは従来より速く広がります。しかも厄介なのは、コードベースが一見きれいで、テストも通っているため、問題が静かに潜伏することです。Addy は、「テストが通った」と「自分たちがそれを理解している」は別物だと強く警告しています。  ￼

[https://www.oreilly.com/radar/comprehension-debt-the-hidden-cost-of-ai-generated-code/:embed]

ここで spec-only の限界が出ます。自然言語仕様がどれほど精密でも、実装には境界条件、例外処理、データ構造の選択、性能トレードオフ、既存コードベースとの接続、UI のふるまい、運用上の抜け道といった暗黙の設計判断が入り込みます。「巨大な仕様を丸ごと投げるな、スマートな仕様にしろ」と言うのはそのためです。仕様は必要ですが、自然言語仕様だけで実装上の暗黙的な判断を封じ込めることには限界がある。もし本当に封じ込めようとすると、その仕様はどんどんコードに似ていきます。すると何が起きるか。コードを目を皿にして確認していた工数が、今度は文書を目を皿にして確認する工数へ“移動”する。これは効率化ではなく、レビュー対象の置換です。  ￼

marmelab の批判は、まさにこの痛点を突いています。そこでは、SDD は実装前に重い設計を積み上げる形で Markdown を大量に生み、レビュー負荷を減らすどころか、むしろ二重化する。と批判されています。さらに、新規開発では多少効いたとしても、大きな既存コードベースではほとんど役に立たないとまで言っています。このような場合の画面要件や振る舞い要件の列挙よりも既存実装の中に埋め込まれた依存関係や制約・暗黙の設計判断にあるからでしょう。言い方はかなり強いですが、論点自体は無視できません。仕様が厚くなるほど安心できる、とは限らないのです。むしろ「読んだ気になれる」ことの方が危ない。  ￼

[https://marmelab.com/blog/2025/11/12/spec-driven-development-waterfall-strikes-back.html:embed]

ただし、ここで「では仕様は要らない」と飛ぶのも短絡です。GitHub も Addy も共通して、AI エージェントに対しては、明確で、過不足なく、構造化された仕様が必要だと述べています。理由は単純で、AI は指示の行間を読まないペアプログラマーとして扱うべきであり、曖昧な指示から無数の暗黙的な要件を正しく推測することはできないからです。だから仕様は必要です。けれど、それは万能の代理人ではなく、AI が勝手に危険な補完をしないために必要なだけ具体的であるべきものです。長さではなく、境界条件と検証可能性が大事です。  ￼

# あるべき仕様駆動開発とはどういうものなのか

ここまで来ると、仕様駆動開発のあるべき姿はだいぶ絞れます。仕様は必要です。だが、仕様だけでは足りません。仕様は SSoT になりうる。だが、それは「他のものを見なくてよい」という意味ではありません。仕様は AI に仕事をさせる入口であり、品質保証や理解の代用品ではない。この三つを同時に言える形でなければ、筋の通った SDD にはなりません。

この点で、[SDD 論文](https://arxiv.org/html/2602.00180v1)が示す “Golden Rule” はかなり重要です。

> The Golden Rule: Use the minimum level of specification rigor that removes ambiguity for your context. Spec-first for AI-assisted initial development; spec-anchored for long-lived production systems; spec-as-source only when generation tooling is mature and trusted.

要するに、「あなたの文脈で曖昧さを除去するのに必要最低限の厳密さで仕様を書け」ということです。AIが補助する初期開発なら spec-first で十分なこともある。長く運用する本番システムなら spec-anchored が必要になる。spec-as-source はコード生成ツールが十分に成熟し信頼できる場合に限られる。逆に言えば、何でもかんでも spec-as-source に寄せる必要はないし、自然言語仕様を無限に厚くすればよいわけでもない。

GitHub の記事を入れると、この話はさらに具体的になります。仕様を起点にして、計画へ落とし、タスクへ割り、各段階で検証し、見落としや漏れを見つけたら戻って直す。つまり、仕様は凍結文書ではなく、実装と検証を通じて更新され続ける living document です。何かおかしければ仕様に戻り、複雑になったら整理し、タスクが大きければ細分化する。ここでの主戦場は、実装後に Word を赤入れすることではなく、仕様そのものを変更管理の中心に置くことです。  ￼

そして、この運用を成立させる条件が、人間の理解責任を消さないことです。要求や edge case は、最初から全部そこにあるわけではありません。実装して、つないで、壊して、レビューして、初めて出てくるものがあります。Addy の理解負債論が言っているのは、まさにここです。AI で実装速度が上がっても、システム全体の理解、つまり「このシステムはなぜこの形なのか」「どの部分が根幹を支えているのか」「どこを変えると全体にどう響くのか」という感覚は、自動で配布されません。だから人間がコードを読む能力は、なくならないどころか、むしろ希少資源になります。AI 時代のエンジニアリングで残る仕事は、全文字を手打ちすることではなく、仕様と実装の差分を見抜き、変更影響を判断し、理解負債が積み上がらないようにすることです。  ￼

この意味で、あるべき仕様駆動開発とは「AIに全部任せる方法」ではありません。AI に委譲しやすいように仕様を外化しつつ、その仕様を更新可能な運用対象として扱い、人間が理解・検証・修正のループを持ち続ける方法です。仕様は入口です。出口ではありません。人間の理解は保険です。飾りではありません。

# まとめ

黙れ小僧！仕様駆動開発に俺たちの仕様を救えるか！
要件定義で固めたはずの仕様を、基本設計で言い換え、詳細設計で分解し、実装で解釈し、テストで真意を問い直す。その長い工程のあいだで、誰も嘘をついておらぬのに、少しずつ意味がズレていく！
人間にもAIにも読みやすい仕様を書けと言うが、業務都合も、過去資産も、レビュー観点も、承認の段取りも、一枚の仕様書だけでは抱えきれぬ！そもそもお前ら人類は判断基準の言語化すらできぬではないか！
Wordにもなりきれず、Markdownで完結できず、チケットにもExcelにも散っていく、哀れで愛しい我らの仕様だ！
お前にこの仕様を救えるか！

完全には救えません。仕様は人間社会の産物なので、承認経路も、政治も、既存資産も、言い換えも、現場の苦心も抱えています。仕様駆動開発が突然それを浄化してくれるわけではありません。marmelab が警戒したように、やり方を誤れば SDD は重い文書駆動に戻り、読むだけで一日が終わる未来を招きます。Addy が警告したように、AI が大量にコードを書いても、人間が理解しなければ理解負債は静かに膨らみます。

それでも、仕様駆動開発が俺たちの開発を少しは救う可能性はあります。救済の形は、「仕様書を一冊きれいにすること」ではありません。AI に渡す意図を明示し、勝手な補完を減らし、実装と検証を通じて仕様を更新し、人間の理解責任を散らさずに保つことです。GitHub が言うように、今は「コードが SSoT」から「意図が SSoT」へ重心が移りつつあります。けれど、その意図を保守するのは、結局のところ人間です。仕様駆動開発が救うのは仕様そのものではなく、仕様をめぐる意思決定の秩序です。  ￼

だから、私なりの結論はこうです。仕様駆動開発は、俺たちの仕様を魔法のように救いません。しかし、仕様を凍結文書ではなく、AI と人間が共同で扱う運用対象に変えることで、ズレを早く見つけ、理解負債を小さくし、開発を少しまともにすることはできます。そのために必要なのは、spec-only の夢を見ることではなく、仕様をどの粒度で、どの段階で、誰が、どう更新するのかを設計することです。

要するに、仕様駆動開発の本質は「仕様を増やすこと」ではありません。仕様を運用することです。
